SLIDE 2 — SOLUTION PURPOSE
===============================================================
Header: What Validhater Is For

Body content (as bullet points):
- Validhater is an internal LQA (Linguistic Quality Assurance) tool built to evaluate Norwegian (NB-NO) translations produced for the Amazon TDT (Translation Data for Training) program.
- It uses a large language model (Claude, via AWS Bedrock) as an automated linguistic evaluator — not as a translator. It never generates or modifies translations; it only scores and critiques existing human-produced or human-post-edited text.
- Purpose: catch translation errors (mistranslation, omission, grammar, terminology, locale-formatting issues) at 100% segment coverage, at a scale and speed manual QA sampling cannot match.
- It was introduced to help the language team meet a tight delivery timeline on a large-volume Amazon program while maintaining a "quality first" bar, following prior performance issues on the account (ASINs) that made this program a second chance for the team.
- It supports three check modes:
  1. PU (Publish check) — did the linguist correctly decide to publish this string as-is?
  2. PE (Post-edit check) — did the linguist correctly decide to send this string for post-editing?
  3. PEQA (Post-edit QA) — final quality score on the linguist's post-edited output, flagging anything scoring below a 95/100 threshold for rework.
- Output: for every batch, an Excel report per segment (score, severity, error category, reasoning) plus a per-linguist performance summary, used by Language Leads to decide what needs correction before final delivery.
- Human accountability is preserved throughout: Validhater flags and explains; Language Leads review every flagged segment and make the final call. The AI does not autonomously alter or approve any deliverable.

SLIDE 3 — SOLUTION DESIGN / ARCHITECTURE
===============================================================
Header: How Validhater Is Built

Body content:
- Interface: Streamlit web application. Language Leads access it via a browser; no local install required.
- Hosting: Streamlit Community Cloud (Streamlit Cloud), connected to a private GitHub repository. Deploys automatically from the main branch.
- Core logic: Python. Main modules: streamlit_app.py (UI layer), va_evaluator.py (evaluation engine, Excel report generation, reference-data loading).
- AI model access: AWS Bedrock (Anthropic Claude Sonnet 4.6, cross-region inference). The tool calls Bedrock's native API — not the AI vendor's direct consumer API — so all model inference happens inside AWS infrastructure under Acclaro/Amazon's existing AWS agreements, not a third-party AI vendor's own servers.
- Authentication: username/password login gate stored in Streamlit's encrypted secrets store (st.secrets), checked at session start. Currently shared credentials across Language Leads — flagged in this review as an action item to move to individual accounts (see Slide 6).
- Secrets management: AWS Bedrock bearer token and user passwords are held in Streamlit Cloud's secrets manager (equivalent to environment variables, encrypted at rest, never committed to source control). No secrets are hardcoded in the codebase.
- Reference data embedded in the evaluation logic (all static, no external calls at runtime beyond the Bedrock API):
  - Amazon NB-NO style guide rules, encoded directly into the evaluator's system prompt.
  - Amazon Global Term Base (EN→NB-NO term pairs), loaded from a local reference file and matched against each segment before evaluation.
  - Evaluation rubric refined through 7 rounds of Language Lead feedback (scoring scale, error taxonomy, edge-case handling).
- Data handling: uploaded files (.xlsx exports from the TMS) are processed in-memory for the duration of the session only. No segment content, translation text, or client data is written to persistent storage, a database, or logged externally. Data exists only in the browser session and in the downloaded report the user generates.
- No data retention: once a Language Lead closes their session or logs out, in-memory data is cleared. The Excel report is the only artifact that persists, and it is downloaded directly to the user's machine — Validhater itself does not store completed reports.
- Dependencies (minimal, 3 packages): boto3 (AWS SDK), streamlit (web framework), openpyxl (Excel read/write).

SLIDE 4 — FULL CYCLE / WORKFLOW
===============================================================
Header: End-to-End Workflow

Use this as the basis for a step-by-step flow diagram (left to right or top to bottom), 7 stages. Each stage below = one diagram node. Draw arrows in sequence 1 → 2 → 3 → 4 → 5 → 6 → 7, with stage 6 shown as a decision/loop point (feedback can route back into the cycle before final submission — see note after step 6).

1. Source prep (Amazon / ATMS)
   Amazon prepares source files and applies machine translation (MT) on the source content inside ATMS (Amazon's translation management system).

2. Handoff to WOL
   The ATMS connector submits the resulting bilingual files (mxliff format) to Acclaro's TMS, WOL.

3. Human validation + post-editing (WOL)
   Linguists and validators work the content inside WOL: validating MT output (publish or send-to-post-edit decisions) and post-editing where required.

4. Export to Validhater
   Language Leads export the post-edited / validated files from WOL and upload them into Validhater (.xlsx format).

5. Automated LQA (Validhater)
   Validhater runs the AI evaluation (PU, PE, or PEQA mode depending on the check needed) across 100% of uploaded segments and generates an LQA report (Excel) with scores, flagged issues, and reasoning per segment.

6. Language Lead review (human-in-the-loop)
   Language Leads review the LQA report and:
     a. Apply necessary corrections directly in WOL.
     b. Provide feedback to Ahmet (tool owner) if they disagree with an AI-flagged result — this feedback is used to refine the evaluation rubric over time.
     c. Provide feedback to individual linguists where a quality or process issue is identified.

7. Final submission
   The corrected job is submitted back through WOL, completing the cycle.

Diagram note: label stage 5 clearly as "AI-assisted, automated" and stage 6 as "Human decision point" to visually reinforce that the AI never has final say — every AI output is reviewed and actioned by a human before delivery.
